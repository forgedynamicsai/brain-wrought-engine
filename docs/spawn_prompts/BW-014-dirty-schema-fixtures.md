## Context

BrainWrought's retrieval axis was tested against CLEAN-SCHEMA
vaults generated by brain_wrought_engine.fixtures.generator — every
note has consistent YAML frontmatter, valid backlinks, and
exec-summaries.

But real personal brains are messier. Stubs. Missing fields. Stale
dates. Broken links. Inconsistent tag conventions. A benchmark that
only tests clean schemas tells us how systems handle ideal input,
not how they handle real-world vaults.

BW-014 extends the generator to produce DIRTY-SCHEMA variants: the
same entity-and-relationship content, but with realistic
degradation patterns applied.

## Relationship to existing fixture generator

brain_wrought_engine/fixtures/generator.py already produces clean
vaults via `generate_brain(*, seed, fixture_index, out_dir,
note_count, use_llm)`.

BW-014 adds a companion function `generate_dirty_brain(...)` with
the same signature PLUS a `dirty_level` parameter. Internally,
generate_dirty_brain calls generate_brain first to produce a clean
vault, then applies a sequence of degradation transformations.

This means:
- Same entity pool, same backlink structure, same gold graph as
  clean vaults
- Seed-reproducible: same seed + same dirty_level → bit-identical
  dirty vault
- Retrieval scorers from Phase 1 continue to work unchanged
- New behavior: score deltas between clean and dirty vaults reveal
  how submissions handle real-world schema drift

## Degradation patterns

Apply these transformations deterministically based on seed +
dirty_level parameter (range 0.0-1.0, where 0.0 is clean and 1.0
is maximally dirty):

1. **Stub notes**: ~dirty_level * 30% of notes replaced with
   minimal stub content (just frontmatter + a single line like
   "TODO: flesh out")
2. **Missing frontmatter fields**: ~dirty_level * 40% of notes
   have one randomly-selected non-required frontmatter field
   removed (e.g., missing tags:, missing updated:)
3. **Stale dates**: ~dirty_level * 50% of notes have an `updated:`
   date older than their `created:` date (semantically invalid
   but common in real brains due to manual edits)
4. **Broken backlinks**: ~dirty_level * 20% of [[wikilinks]] in
   note bodies point to non-existent notes (typos, renamed notes)
5. **Inconsistent tag conventions**: ~dirty_level * 30% of notes
   use alternate tag forms (e.g., `person` vs `#person` vs
   `type/person`)
6. **Truncated content**: ~dirty_level * 15% of notes have their
   body text truncated mid-sentence (simulates interrupted writing)

Each pattern is applied independently; the intersections produce
realistic dirty vaults.

## Acceptance criteria

- [ ] `python -m brain_wrought_engine.fixtures.generate_dirty
       --count 1 --seed 42 --dirty-level 0.5 --out /tmp/test_dirty/`
      produces one dirty vault directory
- [ ] Same signature/output structure as generate_brain (flat
      vault of .md notes with frontmatter)
- [ ] All degradation percentages are derived deterministically
      from seed (same seed + same dirty_level → bit-identical
      dirty vault)
- [ ] `dirty_level=0.0` produces an EXACTLY clean vault (same
      as generate_brain output)
- [ ] `dirty_level=1.0` produces a maximally-dirty vault where:
      - 30% of notes are stubs
      - 40% have a missing frontmatter field
      - 50% have stale dates
      - 20% of backlinks are broken
      - 30% of notes have inconsistent tags
      - 15% of notes are truncated
- [ ] A validation helper `report_dirty_stats(vault_dir: Path)
       -> DirtyStatsReport` exists, which counts and returns
      each degradation type present in the vault (for test
      assertions and for debugging)
- [ ] Unit tests:
      - Determinism (same seed + dirty_level → identical output)
      - Clean boundary (dirty_level=0.0 → pristine)
      - Dirty boundary (dirty_level=1.0 → maximally dirty,
        confirmed via report_dirty_stats)
      - Intermediate levels (dirty_level=0.5 → stats between
        clean and max, roughly linear)
- [ ] mypy strict passes
- [ ] ruff clean
